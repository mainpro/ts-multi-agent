# Critic 审计架构审查机制详细逻辑

## 1. 数据存储架构

### 1.1 data/reports 目录

#### 目录用途
`data/reports` 目录用于存储各种审查报告，包括智能体审查报告、性能报告和合规报告等。

#### 目录结构
```
data/reports/
├── critic/
│   ├── daily/          # 每日审查报告
│   ├── weekly/         # 每周审查报告
│   └── monthly/        # 月度审查报告
├── performance/        # 性能报告
└── compliance/         # 合规报告
```

#### 写入时机
1. **每日审查报告**：在每日结束时，对当天所有审查过的任务进行汇总分析后生成
2. **每周审查报告**：在每周结束时，对当周所有审查过的任务进行汇总分析后生成
3. **月度审查报告**：在每月结束时，对当月所有审查过的任务进行汇总分析后生成
4. **性能报告**：定期或在系统性能发生变化时生成
5. **合规报告**：定期或在合规检查后生成

#### 报告格式
所有报告均采用 Markdown 格式，便于阅读和分享。

##### 审查报告格式（critic）
审查报告包含以下内容：
1. **报告信息**：报告类型、生成时间、审查任务数、发现问题数、高优先级问题、建议实施率
2. **问题类型分布**：按类型统计问题数量（幻觉、错误、低效等）
3. **严重程度分布**：按严重程度统计问题数量（低、中、高）
4. **主要问题**：列出最常出现的Top N问题，包含描述、出现次数、严重程度、证据
5. **优化建议**：列出最常出现的Top N建议，包含描述、出现次数、优先级、实施步骤
6. **详细分析**：对每个审查任务的详细分析结果

文件命名格式：
- 每日：`YYYY-MM-DD.md`
- 每周：`YYYY-WW.md`（如 2026-W15）
- 月度：`YYYY-MM.md`（如 2026-04）

#### 报告生成流程
1. **收集分析数据**：收集指定时间段内的所有 Critic 分析结果
2. **准备报告数据**：
   - 统计总任务数、总问题数、高优先级问题数
   - 分析问题类型分布
   - 分析严重程度分布
   - 提取主要问题（Top N）
   - 提取优化建议（Top N）
3. **生成 Markdown 内容**：根据报告数据生成格式化的 Markdown 内容
4. **写入文件**：将 Markdown 内容写入相应的报告文件
5. **记录日志**：记录报告生成日志

### 1.2 data/knowledge 目录

#### 目录用途
`data/knowledge` 目录用于存储优化建议知识库，包括 Critic 生成的所有优化建议及其状态跟踪信息。

#### 目录结构
```
data/knowledge/
└── optimization-suggestions.json    # 优化建议数据库
```

#### 写入时机
1. **创建建议**：当 Critic 生成新的优化建议时写入
2. **更新建议状态**：当建议状态发生变化时更新（待处理 → 进行中 → 已实施）
3. **更新建议内容**：当建议内容需要修改时更新
4. **删除建议**：当建议过期或不再需要时删除
5. **导入建议**：批量导入外部建议时

#### 数据格式
建议采用 JSON 格式存储，单个建议的结构如下：

```typescript
{
  "id": "suggestion-1776395004269-task-1-vlek4zdff",  // 建议唯一标识符
  "description": "优化 ees-qa 技能执行",              // 建议描述
  "type": "skill",                                     // 建议类型：skill/agent/system/process
  "priority": "medium",                                // 优先级：low/medium/high/critical
  "status": "pending",                                 // 状态：pending/in_progress/implemented/rejected/deferred
  "createdAt": "2026-04-20T03:03:27.566Z",             // 创建时间
  "updatedAt": "2026-04-20T03:03:27.566Z",             // 最后更新时间
  "implementationSteps": [                            // 实施步骤
    "审查技能文档和提示词",
    "优化工具调用参数",
    "增加输入验证",
    "添加更多错误处理"
  ],
  "relatedSkills": ["ees-qa"],                         // 相关技能（可选）
  "relatedAgents": ["sub"],                            // 相关智能体（可选）
  "evidence": "任务执行路径过长，存在冗余步骤",        // 证据（可选）
  "severity": "low",                                   // 严重程度（可选）
  "impact": "提高执行效率，减少资源消耗",              // 影响说明（可选）
  "implementationEstimate": 4,                         // 预计实施时间（小时）（可选）
  "implementedBy": "admin",                            // 实施者（可选，已实施状态时）
  "implementedAt": "2026-04-22T10:15:30.000Z",         // 实施时间（可选，已实施状态时）
  "rejectionReason": "已有类似优化",                   // 拒绝原因（可选，已拒绝状态时）
  "deferralReason": "资源不足，延迟处理",              // 延迟原因（可选，已延迟状态时）
  "deferralDate": "2026-05-01T00:00:00.000Z"           // 延迟日期（可选，已延迟状态时）
}
```

#### 建议状态流转
```
pending（待处理）
  ├─→ in_progress（进行中）
  │     └─→ implemented（已实施）
  ├─→ rejected（已拒绝）
  └─→ deferred（已延迟）
        └─→（可重新转为 pending 或 in_progress）
```

#### 主要操作
1. **创建建议**：自动生成或手动创建优化建议
2. **查询建议**：
   - 按状态查询
   - 按优先级查询
   - 按类型查询
   - 按技能查询
3. **更新建议**：
   - 更新建议状态
   - 更新建议内容
4. **统计分析**：
   - 统计总数
   - 按状态统计
   - 按优先级统计
   - 按类型统计
   - 计算实施率
   - 计算平均年龄
5. **导出导入**：
   - 导出建议
   - 批量导入建议
6. **清理过期**：清理已实施或已拒绝且超过指定天数的建议

#### 数据持久化流程
1. **加载数据**：系统启动时从文件加载所有建议到内存 Map 中
2. **内存操作**：所有增删改操作先在内存中执行
3. **保存到文件**：每次修改后自动将所有建议序列化为 JSON 格式保存到文件
4. **日期转换**：加载时将 JSON 中的日期字符串转换为 Date 对象，保存时再转回字符串

## 2. 审查流程

Critic 智能体的审查流程包含以下几个关键步骤：

### 2.1 初始化阶段

1. **加载专业技能**：CriticAgent 初始化时，会扫描并加载专业技能目录中的所有技能
2. **构建技能注册表**：将专业技能注册到 ProfessionalSkillRegistry 中，便于后续调用

### 2.2 任务审查流程

对于每个待审查的任务，CriticAgent 执行以下步骤：

#### 1. 错误分析（如果任务失败）

- **检查任务状态**：如果任务状态为 `failed` 且包含错误信息
- **调用错误分析技能**：使用 `error-analyzer` 专业技能分析错误
- **提取错误信息**：从任务中提取错误类型、消息和代码
- **分析根因**：使用 LLM 分析错误的根本原因
- **评估影响**：评估错误对系统的影响程度
- **生成解决方案**：基于错误类型生成具体的解决方案

#### 2. 幻觉检测（如果任务成功）

- **检查任务结果**：如果任务状态为 `completed` 且包含结果
- **调用幻觉检测技能**：使用 `hallucination-detector` 专业技能检测幻觉
- **分析输出内容**：分析智能体的输出内容
- **对比工具结果**：与工具调用的实际结果进行对比
- **检测逻辑矛盾**：检查输出中的逻辑一致性
- **验证事实准确性**：验证关键事实的准确性
- **生成改进建议**：基于检测结果生成改进建议

#### 3. 执行路径分析

- **检查执行路径**：如果任务包含执行路径信息
- **分析失败步骤**：识别执行过程中的失败步骤
- **分析路径长度**：检查执行路径是否过长，可能存在冗余步骤
- **生成优化建议**：针对执行路径中的问题生成优化建议

#### 4. 错误模式分析

- **检查错误历史**：如果任务包含错误历史
- **识别重复错误**：分析错误历史，识别重复出现的错误模式
- **生成针对性建议**：基于错误模式生成针对性的解决方案

#### 5. 通用优化建议

- **基于技能类型**：根据任务使用的技能类型，生成通用的优化建议
- **审查技能文档**：建议审查技能文档和提示词
- **优化工具调用**：建议优化工具调用参数
- **增加输入验证**：建议增加输入验证
- **添加错误处理**：建议添加更多错误处理

#### 6. 置信度计算

- **基础置信度**：设置基础置信度为 0.5
- **根据问题数量调整**：如果发现问题，增加置信度
- **根据解决方案数量调整**：如果生成了解决方案，增加置信度
- **限制范围**：确保置信度在 0-1 之间

### 2.3 批量审查

CriticAgent 还支持批量审查多个任务，通过 `reviewTasks` 方法实现，对每个任务执行相同的审查流程。

## 3. 专业技能实现

### 3.1 幻觉检测技能

#### 核心职责

- **事实一致性检查**：验证智能体输出与工具调用结果的一致性
- **逻辑连贯性分析**：检测输出中的逻辑矛盾
- **事实准确性验证**：通过外部知识库验证事实

#### 检测维度

| 维度 | 描述 | 检测方法 |
|------|------|----------|
| 事实幻觉 | 编造不存在的事实 | 与工具调用结果对比 |
| 逻辑幻觉 | 前后矛盾的陈述 | 逻辑一致性分析 |
| 数据幻觉 | 编造具体数据 | 数据源验证 |

#### 检测流程

1. **输入分析**：分析智能体的输出内容
2. **工具调用结果对比**：与工具调用的实际结果进行对比
3. **逻辑分析**：检查输出中的逻辑一致性
4. **事实验证**：验证关键事实的准确性
5. **结果生成**：生成检测结果和改进建议

### 3.2 错误分析技能

#### 错误分类

| 错误类型 | 描述 | 常见场景 |
|---------|------|---------|
| API_ERROR | API 调用失败 | 网络问题、权限问题 |
| TOOL_ERROR | 工具执行错误 | 参数错误、超时 |
| LOGIC_ERROR | 逻辑错误 | 推理错误、路径选择错误 |
| HALLUCINATION | 幻觉 | 编造事实 |
| TIMEOUT | 超时错误 | 执行时间过长 |
| MEMORY_ERROR | 内存错误 | 内存使用过高 |

#### 分析流程

1. **错误信息提取**：从任务日志中提取错误详情
2. **错误模式识别**：匹配已知错误模式
3. **根因分析**：分析错误的根本原因
4. **影响评估**：评估错误对系统的影响
5. **解决方案生成**：生成具体的解决方案

#### 根因分析方法

1. **故障树分析**：从错误现象出发，逐步分析可能的原因
2. **模式匹配**：匹配已知的错误模式，利用历史数据和经验
3. **相关性分析**：分析错误与其他事件的相关性，识别触发条件

## 4. 执行专业技能的机制

CriticAgent 通过 `executeProfessionalSkill` 方法执行专业技能：

1. **构建提示词**：根据专业技能的定义和输入数据构建提示词
2. **调用 LLM**：使用 `LLMClient.generateText` 方法调用 LLM
3. **解析结果**：从 LLM 响应中提取 JSON 格式的分析结果
4. **处理错误**：如果执行失败，记录详细的错误信息
5. **返回结果**：返回专业技能的执行结果

## 5. 与现有系统的集成

### 5.1 集成方式

- **任务完成监听器**：在任务完成后自动触发审查
- **定期审查**：定期对历史任务进行批量审查
- **报告生成**：定期生成审查报告

### 5.2 数据流

1. **任务执行**：智能体执行任务，产生执行结果或错误
2. **触发审查**：任务完成后触发 Critic 审查
3. **执行审查**：Critic 执行审查流程，生成分析结果
4. **存储结果**：将分析结果存储到任务对象中
5. **生成建议**：基于分析结果生成优化建议
6. **生成报告**：定期生成审查报告

## 6. 配置和部署

### 6.1 配置选项

- **专业技能目录**：默认为 `./professional-skills/`
- **审查触发机制**：可配置为任务完成后自动触发或定期触发
- **审查频率**：可配置审查的频率和时间
- **报告生成频率**：可配置报告生成的频率（每日、每周、每月）

### 6.2 部署建议

- **独立部署**：Critic 智能体可作为独立服务部署
- **后台执行**：审查过程应在后台执行，不影响主系统性能
- **资源配置**：根据系统规模和任务量配置适当的资源
- **监控**：监控 Critic 智能体的运行状态和性能

## 7. 错误处理和容错

### 7.1 错误处理机制

- **专业技能执行失败**：如果专业技能执行失败，记录详细错误信息并继续执行审查
- **LLM 调用失败**：如果 LLM 调用失败，使用降级方案继续审查
- **审查过程失败**：如果审查过程失败，确保不影响主系统的正常运行

### 7.2 降级方案

- **专业技能不可用**：如果专业技能不可用，使用内置的简单分析逻辑
- **LLM 不可用**：如果 LLM 不可用，使用基于规则的分析方法
- **系统资源不足**：如果系统资源不足，降低审查频率或范围

## 8. 性能优化

### 8.1 性能考虑

- **异步执行**：审查过程应异步执行，不阻塞主系统
- **批处理**：对多个任务进行批处理，减少 LLM 调用次数
- **缓存**：缓存专业技能的执行结果，避免重复分析
- **资源限制**：限制审查过程的资源使用，确保不影响主系统

### 8.2 优化策略

- **智能调度**：根据系统负载智能调度审查任务
- **优先级排序**：优先审查重要任务或失败任务
- **增量分析**：只分析任务的变化部分，减少分析量
- **并行处理**：对多个任务进行并行审查，提高效率

2. **专业的技能**：使用专业的审查技能，提供深度分析
3. **智能的建议**：基于分析结果生成具体的优化建议
4. **持续的改进**：通过定期审查和报告，推动系统的持续改进
5. **非侵入式设计**：采用观察者模式，不影响现有系统流程
6. **可扩展的架构**：支持添加新的专业技能和审查维度

通过 Critic 审计架构，智能体系统能够不断自我进化，减少错误重复率，提高执行效率，为用户提供更可靠、更高效的智能体服务。
