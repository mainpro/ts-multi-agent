## P0 — 紧急（安全加固与可靠性）

### P0-1: 实施 allowedTools 权限过滤

**对应缺点**: #5 — SubAgent 的 allowedTools 字段未生效

#### 问题分析

当前 `SkillMetadata` 定义了 `allowedTools?: string[]` 字段，但 `SubAgent.executeSkill()` 方法中注册工具时，直接使用了 `this.toolRegistry.list()` 获取全部工具，完全没有根据技能的 `allowedTools` 进行过滤。这意味着任何技能的 SubAgent 都可以调用所有已注册的工具（包括 bash、write、edit 等高危工具），存在严重的安全隐患。

#### 技术方案设计

在 `SubAgent.executeSkill()` 方法中，加载技能后根据 `skill.allowedTools` 过滤可用工具列表。如果技能未定义 `allowedTools`，则使用默认安全工具白名单（仅包含只读工具）。

设计要点：
1. 定义默认安全工具白名单 `DEFAULT_SAFE_TOOLS`
2. 在 `executeSkill` 中根据 `skill.allowedTools` 过滤工具
3. 未配置 `allowedTools` 时使用默认白名单（保守策略）
4. 在日志中记录被过滤掉的工具，便于审计

#### 核心代码示例

```typescript
// src/agents/sub-agent.ts

// 默认安全工具白名单（仅只读工具）
const DEFAULT_SAFE_TOOLS = new Set([
  'context-get',
  'context-get-all',
  'conversation-get',
  'read',
  'glob',
  'grep',
  'vision-analyze',
]);

private async executeSkill(
  requirement: string,
  skill: Skill,
  params?: Record<string, unknown>,
  sessionId?: string,
  userId?: string,
  questionHistory?: Array<{...}>,
  signal?: AbortSignal
): Promise<SkillExecutionResult> {
  const skillRootDir = './skills/' + skill.name;
  const absoluteSkillRootDir = require('path').resolve(skillRootDir);
  const systemPrompt = buildSubAgentPrompt(skill.body, absoluteSkillRootDir, params, questionHistory);

  // ===== 新增：根据 allowedTools 过滤工具 =====
  const allTools = this.toolRegistry.list();
  let allowedToolNames: Set<string>;

  if (skill.allowedTools && skill.allowedTools.length > 0) {
    // 技能显式声明了允许使用的工具
    allowedToolNames = new Set(skill.allowedTools);
    console.log(`[SubAgent] 技能 ${skill.name} 限制工具: ${skill.allowedTools.join(', ')}`);
  } else {
    // 未声明时使用默认安全白名单
    allowedToolNames = DEFAULT_SAFE_TOOLS;
    console.log(`[SubAgent] 技能 ${skill.name} 未声明 allowedTools，使用默认安全白名单`);
  }

  const filteredTools = allTools.filter((tool: Tool) => {
    if (!allowedToolNames.has(tool.name)) {
      console.log(`[SubAgent] ⚠️ 工具 ${tool.name} 被过滤（不在 allowedTools 中）`);
      return false;
    }
    return true;
  });

  if (filteredTools.length === 0) {
    console.warn(`[SubAgent] ⚠️ 技能 ${skill.name} 没有可用工具！`);
  }

  const tools = filteredTools.map((tool: Tool) => ({
    name: tool.name,
    description: tool.description,
    parameters: {
      type: 'object',
      properties: tool.parameters || {},
      required: tool.required || [],
    },
  }));
  // ===== 过滤逻辑结束 =====

  // ... 后续代码不变
}
```

### P0-2: 添加敏感文件保护

**对应缺点**: #7 — 无路径安全检查

#### 问题分析

当前所有文件操作工具（read、write、edit）和 bash 工具都没有对敏感路径进行保护。SubAgent 可以直接读取 `.env`、`.ssh/id_rsa`、`/etc/passwd` 等敏感文件，也可以通过 bash 执行 `rm -rf /` 等破坏性命令。

#### 技术方案设计

1. 创建 `PathGuard` 工具类，提供路径安全检查
2. 定义敏感路径黑名单（系统级 + 项目级）
3. 在 `BaseTool.execute()` 或各工具的 `execute()` 中集成路径检查
4. 对于敏感路径操作，返回明确的拒绝信息而非执行

#### 核心代码示例

```typescript
// src/security/path-guard.ts

export class PathGuard {
  // 系统级敏感路径（绝对禁止访问）
  private static readonly SYSTEM_SENSITIVE_PATTERNS = [
    /\/\.ssh\//,
    /\/\.aws\//,
    /\/\.gnupg\//,
    /\/\.env/,
    /\/etc\/(shadow|passwd|sudoers)/,
    /\/proc\//,
    /\/sys\//,
  ];

  // 项目级敏感路径（需要额外确认，但当前版本直接拒绝）
  private static readonly PROJECT_SENSITIVE_PATTERNS = [
    /\.env$/,
    /\.env\./,
    /credentials/i,
    /secret/i,
    /private[_-]?key/i,
  ];

  /**
   * 检查路径是否安全
   * @returns { safe: boolean, reason?: string }
   */
  static checkPath(filePath: string): { safe: boolean; reason?: string } {
    const normalizedPath = require('path').resolve(filePath);

    // 检查系统级敏感路径
    for (const pattern of this.SYSTEM_SENSITIVE_PATTERNS) {
      if (pattern.test(normalizedPath)) {
        return { safe: false, reason: `系统敏感路径，禁止访问: ${normalizedPath}` };
      }
    }

    // 检查项目级敏感路径
    for (const pattern of this.PROJECT_SENSITIVE_PATTERNS) {
      if (pattern.test(normalizedPath)) {
        return { safe: false, reason: `敏感文件，禁止访问: ${normalizedPath}` };
      }
    }

    return { safe: true };
  }

  /**
   * 检查 bash 命令是否安全（基础检查）
   */
  static checkBashCommand(command: string): { safe: boolean; reason?: string } {
    const dangerousPatterns = [
      /rm\s+(-[rf]+\s+)?\//,       // rm -rf /
      /mkfs\./,                      // 格式化文件系统
      /dd\s+if=/,                    // dd 磁盘操作
      />\s*\/dev\//,                 // 重定向到设备文件
      /chmod\s+777/,                 // 设置全局可写权限
      /curl.*\|\s*(ba)?sh/,          // 远程脚本执行
      /wget.*\|\s*(ba)?sh/,          // 远程脚本执行
    ];

    for (const pattern of dangerousPatterns) {
      if (pattern.test(command)) {
        return { safe: false, reason: `危险命令模式被拦截: ${command}` };
      }
    }

    return { safe: true };
  }
}
```

### P0-3: 任务状态持久化

**对应缺点**: #12 — TaskQueue 纯内存，崩溃丢失

#### 问题分析

当前 `TaskQueue` 使用 `Map<string, Task>` 存储所有任务状态，完全在内存中。进程崩溃或重启后，所有进行中的任务状态丢失，包括：
- 正在执行的 SubAgent 任务
- 等待依赖的任务
- 已完成但未返回给用户的结果

#### 技术方案设计

采用轻量级的文件持久化方案（JSON 文件），在关键状态变更时写入磁盘：

1. 定义 `TaskQueueStorage` 接口，支持文件存储
2. 在 `addTask`、`completeTask`、`failTask` 时触发持久化
3. 启动时从文件恢复未完成的任务状态
4. 使用防抖（debounce）避免频繁写盘

#### 核心代码示例

```typescript
// src/task-queue/storage.ts

import { Task } from '../types';
import * as fs from 'fs';
import * as path from 'path';

export class TaskQueueStorage {
  private filePath: string;
  private writeTimer: ReturnType<typeof setTimeout> | null = null;
  private readonly DEBOUNCE_MS = 1000; // 1秒防抖

  constructor(dataDir: string = 'data/task-queue') {
    this.filePath = path.join(dataDir, 'tasks.json');
    // 确保目录存在
    fs.mkdirSync(dataDir, { recursive: true });
  }

  /**
   * 保存任务快照（防抖）
   */
  save(tasks: Map<string, Task>): void {
    if (this.writeTimer) {
      clearTimeout(this.writeTimer);
    }
    this.writeTimer = setTimeout(() => {
      this.persist(tasks);
    }, this.DEBOUNCE_MS);
  }

  private persist(tasks: Map<string, Task>): void {
    try {
      const snapshot = Array.from(tasks.entries()).map(([id, task]) => ({
        id,
        ...task,
      }));
      fs.writeFileSync(this.filePath, JSON.stringify(snapshot, null, 2), 'utf-8');
    } catch (error) {
      console.error('[TaskQueueStorage] 持久化失败:', error);
    }
  }

  /**
   * 加载任务快照
   */
  load(): Map<string, Task> {
    const tasks = new Map<string, Task>();
    try {
      if (!fs.existsSync(this.filePath)) {
        return tasks;
      }
      const data = fs.readFileSync(this.filePath, 'utf-8');
      const snapshot = JSON.parse(data);
      for (const item of snapshot) {
        const { id, ...task } = item;
        // 恢复 Date 对象
        if (task.createdAt) task.createdAt = new Date(task.createdAt);
        if (task.startedAt) task.startedAt = new Date(task.startedAt);
        if (task.completedAt) task.completedAt = new Date(task.completedAt);
        // 将 running 状态重置为 pending（进程重启后无法恢复执行中的任务）
        if (task.status === 'running') {
          task.status = 'pending';
          console.log(`[TaskQueueStorage] 任务 ${id} 状态从 running 重置为 pending`);
        }
        tasks.set(id, task);
      }
      console.log(`[TaskQueueStorage] 恢复了 ${tasks.size} 个任务`);
    } catch (error) {
      console.error('[TaskQueueStorage] 加载失败:', error);
    }
    return tasks;
  }
}
```

### P0-4: 错误恢复增强

**对应缺点**: #13 — 错误恢复机制薄弱

#### 问题分析

当前系统的错误恢复仅限于 MainAgent 的 `monitorAndReplan`（最多 3 次 replan），且 replan 是粗粒度的——整个计划重新生成。缺乏针对不同错误类型的细粒度恢复策略。

参考 Claude Code 的多层次 fallback：
- `prompt_too_long` → Context Collapse drain → Reactive Compact → 错误
- `max_output_tokens` → 8k→64k 重试 → 多轮继续
- `overloaded_error` → 指数退避重试

#### 技术方案设计

在 LLMClient 层面增加细粒度错误恢复策略：

1. 定义 `ErrorRecoveryStrategy` 映射表
2. 对 `prompt_too_long` 错误：自动触发上下文压缩后重试
3. 对 `overloaded` 错误：延长退避时间后重试
4. 对 `max_tokens` 错误：增大 max_tokens 后重试
5. 在 SubAgent 层面增加工具执行错误分类和自动降级

#### 核心代码示例

```typescript
// src/llm/error-recovery.ts

import { LLMError, LLMErrorType } from './index';
import { AutoCompactService } from '../memory/auto-compact';

export interface RecoveryAction {
  strategy: string;
  description: string;
  execute: () => Promise<boolean>; // 返回是否恢复成功
}

export class ErrorRecoveryManager {
  private compactService: AutoCompactService;
  private maxRecoveryAttempts: number = 2;

  constructor(compactService: AutoCompactService) {
    this.compactService = compactService;
  }

  /**
   * 根据错误类型生成恢复策略
   */
  getRecoveryActions(error: LLMError, context: {
    messages?: any[];
    currentMaxTokens?: number;
  }): RecoveryAction[] {
    const actions: RecoveryAction[] = [];

    switch (error.type) {
      case 'RATE_LIMIT':
        actions.push({
          strategy: 'extended_backoff',
          description: '延长退避时间后重试（10s → 30s → 60s）',
          execute: async () => true, // 由调用方控制重试间隔
        });
        break;

      case 'TIMEOUT':
        actions.push({
          strategy: 'increase_timeout',
          description: '增加超时时间后重试',
          execute: async () => true,
        });
        break;

      case 'API_ERROR':
        if (error.message.includes('prompt_too_long') || error.message.includes('context_length')) {
          actions.push({
            strategy: 'context_collapse',
            description: '压缩上下文后重试',
            execute: async () => {
              if (context.messages && this.compactService) {
                const compacted = await this.compactService.autoCompact(context.messages);
                context.messages.length = 0;
                context.messages.push(...compacted);
                return true;
              }
              return false;
            },
          });
        }
        if (error.message.includes('max_output_tokens') || error.message.includes('output_length')) {
          actions.push({
            strategy: 'increase_max_tokens',
            description: '增大 max_tokens 后重试',
            execute: async () => {
              if (context.currentMaxTokens) {
                context.currentMaxTokens = Math.min(context.currentMaxTokens * 4, 65536);
                return true;
              }
              return false;
            },
          });
        }
        break;
    }

    return actions;
  }
}
```

## P1 — 重要（性能优化）

### P1-1: 事件驱动替代轮询

**对应缺点**: #1 — MainAgent 使用轮询等待

#### 问题分析

当前 `MainAgent.waitForCompletion()` 使用 `while(true)` + `sleep(pollInterval)` 轮询 TaskQueue 状态。虽然使用了指数退避（100ms→1s），但仍有以下问题：
- 最坏情况下每次检查间隔 1 秒，任务完成后最多延迟 1 秒才感知
- 轮询期间主线程被阻塞（虽然是非阻塞的 async sleep，但仍占用事件循环）
- 无法实时响应任务完成事件

#### 技术方案设计

在 TaskQueue 中增加 EventEmitter，任务完成/失败时发射事件。MainAgent 通过 Promise 等待事件，而非轮询。

```typescript
private async waitForCompletion(planId: string): Promise<TaskResult> {
  const maxWaitTime = CONFIG.TOTAL_TIMEOUT_MS;
  const startTime = Date.now();

  return new Promise<TaskResult>((resolve) => {
    const checkCompletion = () => {
      const allTasks = this.taskQueue.getAllTasks();
      const planTasks = allTasks.filter(t => t.id.startsWith(planId) && t.skillName);

      if (planTasks.length === 0) {
        resolve({ success: true, data: { planId, results: [] } });
        return;
      }

      const allDone = planTasks.every(t => t.status === 'completed' || t.status === 'failed');
      if (allDone) {
        // 所有任务完成，返回结果
        const failedTasks = planTasks.filter(t => t.status === 'failed');
        if (failedTasks.length === 0) {
          const results = planTasks.filter(t => t.status === 'completed').map(t => ({
            taskId: t.id, skillName: t.skillName, result: t.result,
          }));
          resolve({ success: true, data: { planId, results } });
        } else {
          resolve({ success: false, error: failedTasks[0].error });
        }
        return;
      }
    };

    // 监听任务完成/失败事件
    const onTaskDone = () => checkCompletion();
    this.taskQueue.on('task-completed', onTaskDone);
    this.taskQueue.on('task-failed', onTaskDone);

    // 超时保护
    const timeoutHandle = setTimeout(() => {
      this.taskQueue.off('task-completed', onTaskDone);
      this.taskQueue.off('task-failed', onTaskDone);
      resolve({
        success: false,
        error: { type: 'RETRYABLE', message: `Workflow timeout after ${maxWaitTime}ms`, code: 'TIMEOUT' },
      });
    }, maxWaitTime);

    // 首次检查（可能任务已经完成）
    checkCompletion();

    // 清理函数
    const originalResolve = resolve;
    resolve = ((value: TaskResult) => {
      clearTimeout(timeoutHandle);
      this.taskQueue.off('task-completed', onTaskDone);
      this.taskQueue.off('task-failed', onTaskDone);
      originalResolve(value);
    }) as any;
  });
}
```

### P1-2: 并行工具执行

**对应缺点**: #3 — SubAgent 工具调用串行执行

#### 问题分析

当前 `LLMClient.generateWithTools()` 中，LLM 返回多个 `tool_calls` 时，是逐个执行的（`for...of` 循环 + `await`）。如果 LLM 一次返回 3 个独立的只读工具调用（如 read、glob、grep），它们本可以并行执行，但当前是串行的，浪费了等待时间。

#### 技术方案设计

参考 Claude Code 的 StreamingToolExecutor，将工具分为并发安全和非并发安全两类：
- 并发安全工具（isConcurrencySafe 返回 true）：可以并行执行
- 非并发安全工具：必须串行执行

实现策略：
1. LLM 返回多个 tool_calls 时，先按并发安全性分组
2. 并发安全组使用 `Promise.all()` 并行执行
3. 非并发安全组保持串行执行
4. 两组之间串行（先并发组，再非并发组）

#### 核心代码示例

```typescript
// src/llm/index.ts — 修改 generateWithTools 中的工具执行循环

// 替换原来的 for...of 串行执行
for (const toolCall of message.tool_calls) {
  // ... 原有串行逻辑
}

// 改为分组并行执行：
const safeCalls: typeof message.tool_calls = [];
const unsafeCalls: typeof message.tool_calls = [];

for (const toolCall of message.tool_calls) {
  const tool = toolRegistry?.getTool(toolCall.function.name);
  if (tool && tool.isConcurrencySafe(toolCall.function.arguments)) {
    safeCalls.push(toolCall);
  } else {
    unsafeCalls.push(toolCall);
  }
}

// 并发安全工具并行执行
if (safeCalls.length > 0) {
  console.log(`[LLM] 并行执行 ${safeCalls.length} 个并发安全工具`);
  const safeResults = await Promise.allSettled(
    safeCalls.map(async (toolCall) => {
      const toolName = toolCall.function.name;
      let toolArgs: Record<string, unknown>;
      try {
        toolArgs = JSON.parse(toolCall.function.arguments);
      } catch { toolArgs = {}; }

      let toolResult: string;
      try {
        toolResult = await toolExecutor({ name: toolName, arguments: toolArgs });
      } catch (execError) {
        toolResult = `工具执行错误: ${execError instanceof Error ? execError.message : 'Unknown'}`;
      }

      toolCallsResults.push({ name: toolName, arguments: toolArgs, result: toolResult });
      return { toolCall, toolResult };
    })
  );

  // 将结果添加到 messages
  for (const safeCall of safeCalls) {
    const result = safeResults.find(r => r.status === 'fulfilled' && r.value.toolCall.id === safeCall.id);
    if (result && result.status === 'fulfilled') {
      messages.push({
        role: 'tool',
        content: result.value.toolResult,
        tool_call_id: safeCall.id,
      });
    }
  }
}

// 非并发安全工具串行执行（保持原有逻辑）
for (const toolCall of unsafeCalls) {
  // ... 原有串行执行逻辑
}
```

### P1-3: Token 精确计算

**对应缺点**: #8 — Token 估算使用字符数/4

#### 问题分析

当前 `AutoCompactService.estimateTokens()` 使用 `Math.ceil(contentLength / 4)` 估算 Token 数。这个公式对英文基本准确（1 token ≈ 4 字符），但对中文严重不准（1 个汉字通常 1-2 个 Token，而非 4 个字符对应 1 个 Token）。

实际影响：
- 中文内容被低估约 2-4 倍
- 压缩阈值（167K）可能在实际 Token 已超限时才触发
- 可能导致 API 调用因上下文过长而失败

#### 技术方案设计

使用 `tiktoken` 或轻量级的 `js-tiktoken` 库进行精确 Token 计算。如果不想引入额外依赖，可以使用基于 Unicode 字符类型的改进估算公式。

方案选择：使用 `js-tiktoken`（cl100k_base 编码，兼容 GPT-4/Claude）

#### 核心代码示例

```typescript
// src/memory/token-counter.ts

let encoding: any = null;

async function getEncoding() {
  if (!encoding) {
    try {
      const { encodingForModel } = await import('js-tiktoken');
      // cl100k_base 是 GPT-4 使用的编码，对中英文都有较好支持
      encoding = encodingForModel('gpt-4');
    } catch {
      console.warn('[TokenCounter] js-tiktoken 不可用，回退到估算模式');
      encoding = null;
    }
  }
  return encoding;
}

/**
 * 精确计算 Token 数（使用 tiktoken）
 */
export async function countTokens(text: string): Promise<number> {
  const enc = await getEncoding();
  if (enc) {
    return enc.encode(text).length;
  }
  // 回退：改进的估算公式
  return estimateTokens(text);
}

/**
 * 改进的 Token 估算（区分中英文）
 */
export function estimateTokens(text: string): number {
  let tokens = 0;
  for (const char of text) {
    const code = char.charCodeAt(0);
    if (code > 0x4E00 && code < 0x9FFF) {
      // CJK 统一汉字：每个汉字约 1.5 tokens
      tokens += 1.5;
    } else if (code > 0x3000 && code < 0x303F) {
      // CJK 标点
      tokens += 1;
    } else {
      // ASCII/Latin：约 4 字符 1 token
      tokens += 0.25;
    }
  }
  return Math.ceil(tokens);
}
```

### P1-5: Prompt Cache 优化

**对应缺点**: #9 — 系统提示无缓存优化

#### 问题分析

每次 LLM 调用都会重建完整的 System Prompt，包括技能说明、保底规则等静态内容。这些内容在不同调用间基本不变，但每次都被重新计算和发送，浪费 Token 和 API 成本。

参考 Claude Code 的段落化 System Prompt + 缓存边界设计。

#### 技术方案设计

将 System Prompt 分为静态段落和动态段落：
- **静态段落**: 技能说明、保底规则、基础指令（会话内不变）
- **动态段落**: 用户画像、对话历史、会话变量（每轮变化）

通过标记缓存边界，让 LLM Provider 的 Prompt Caching 机制生效。

#### 核心代码示例

```typescript
// src/prompts/prompt-builder.ts

export class PromptBuilder {
  private staticPromptCache: Map<string, string> = new Map();

  /**
   * 构建带缓存边界的 System Prompt
   */
  static build(
    staticParts: { key: string; content: string }[],
    dynamicParts: string[]
  ): string {
    const staticContent = staticParts.map(p => {
      if (!this.staticPromptCache.has(p.key)) {
        this.staticPromptCache.set(p.key, p.content);
      }
      return this.staticPromptCache.get(p.key)!;
    }).join('\n\n');

    const dynamicContent = dynamicParts.join('\n\n');

    // 使用特殊标记分隔静态和动态区域（部分 Provider 支持）
    return `${staticContent}\n\n<!-- DYNAMIC_CONTEXT_START -->\n${dynamicContent}\n<!-- DYNAMIC_CONTEXT_END -->`;
  }

  /**
   * 清除缓存（技能热重载时调用）
   */
  static clearCache(key?: string): void {
    if (key) {
      this.staticPromptCache.delete(key);
    } else {
      this.staticPromptCache.clear();
    }
  }
}
```

```typescript
export class MetricsCollector {
  private counters: Map<string, number> = new Map();
  private histograms: Map<string, number[]> = new Map();

  incrementCounter(name: string, value: number = 1): void {
    this.counters.set(name, (this.counters.get(name) || 0) + value);
  }

  recordHistogram(name: string, value: number): void {
    if (!this.histograms.has(name)) {
      this.histograms.set(name, []);
    }
    this.histograms.get(name)!.push(value);
  }

  /**
   * 输出 Prometheus 格式指标
   */
  toPrometheus(): string {
    let output = '';
    for (const [name, value] of this.counters) {
      output += `# HELP ${name}\n# TYPE ${name} counter\n${name} ${value}\n`;
    }
    for (const [name, values] of this.histograms) {
      const avg = values.length > 0 ? values.reduce((a, b) => a + b, 0) / values.length : 0;
      const p99 = values.length > 0 ? values.sort()[Math.floor(values.length * 0.99)] : 0;
      output += `# HELP ${name}_avg\n# TYPE ${name}_avg gauge\n${name}_avg ${avg}\n`;
      output += `# HELP ${name}_p99\n# TYPE ${name}_p99 gauge\n${name}_p99 ${p99}\n`;
    }
    return output;
  }
}
```

### P2-2: Hooks 生命周期系统

**对应缺点**: #15 — 缺乏生命周期钩子

#### 问题分析

#### 技术方案设计

定义生命周期事件枚举和 Hook 注册机制：

```typescript
// src/hooks/types.ts

export enum HookEvent {
  BEFORE_INTENT_CLASSIFY = 'before:intent_classify',
  AFTER_INTENT_CLASSIFY = 'after:intent_classify',
  BEFORE_TASK_EXECUTE = 'before:task_execute',
  AFTER_TASK_EXECUTE = 'after:task_execute',
  BEFORE_TOOL_CALL = 'before:tool_call',
  AFTER_TOOL_CALL = 'after:tool_call',
  ON_ERROR = 'on:error',
  ON_FALLBACK = 'on:fallback',
}

export interface HookContext {
  event: HookEvent;
  taskId?: string;
  skillName?: string;
  userId?: string;
  sessionId?: string;
  timestamp: Date;
  data: Record<string, unknown>;
}

export type HookHandler = (context: HookContext) => Promise<void> | void;
```

```typescript
// src/hooks/hook-manager.ts

export class HookManager {
  private hooks: Map<HookEvent, HookHandler[]> = new Map();

  on(event: HookEvent, handler: HookHandler): void {
    if (!this.hooks.has(event)) {
      this.hooks.set(event, []);
    }
    this.hooks.get(event)!.push(handler);
  }

  async emit(event: HookEvent, context: Partial<HookContext>): Promise<void> {
    const handlers = this.hooks.get(event) || [];
    const fullContext: HookContext = {
      event,
      timestamp: new Date(),
      data: {},
      ...context,
    };

    for (const handler of handlers) {
      try {
        await handler(fullContext);
      } catch (error) {
        console.error(`[Hook] ${event} handler error:`, error);
      }
    }
  }
}
```

### P2-3: 技能热重载

**对应缺点**: #16 — SkillRegistry 缺少热重载

#### 问题分析

#### 技术方案设计

#### 核心代码示例

```typescript
// src/skill-registry/index.ts — 添加 watch 方法

import { watch } from 'fs';

private watcher: ReturnType<typeof watch> | null = null;

startWatch(): void {
  const skillsDir = CONFIG.SKILL_DIRECTORY;
  this.watcher = watch(skillsDir, { recursive: true }, (eventType, filename) => {
    if (!filename) return;
    if (filename.endsWith('SKILL.md') || filename.endsWith('skill.md')) {
      console.log(`[SkillRegistry] 检测到技能文件变更: ${filename}`);
      this.rescanSkill(filename);
    }
  });
  console.log(`[SkillRegistry] 已启动技能目录监听: ${skillsDir}`);
}

stopWatch(): void {
  if (this.watcher) {
    this.watcher.close();
    this.watcher = null;
  }
}

private rescanSkill(filename: string): void {
  // 提取技能目录名
  const parts = filename.split('/');
  const skillDirIndex = parts.indexOf('skills');
  if (skillDirIndex === -1 || skillDirIndex + 1 >= parts.length) return;
  const skillName = parts[skillDirIndex + 1];

  // 清除缓存
  this.skillCache.delete(skillName);
  this.metadataCache.delete(skillName);

  // 重新扫描
  const skillDir = path.join(CONFIG.SKILL_DIRECTORY, skillName);
  this.scanSingleSkill(skillDir, skillName);

  console.log(`[SkillRegistry] 技能 ${skillName} 已重新加载`);
}
```

### P3-1: 沙箱隔离

**对应缺点**: #6 — 无沙箱隔离

#### 问题分析

工具直接在宿主环境中执行，bash 工具可以执行任意命令，write/edit 工具可以修改任意文件。

#### 技术方案设计

1. 创建沙箱配置文件（允许读写的目录白名单）
2. bash 工具通过 `bwrap` 包装执行
3. 文件操作工具通过沙箱内的路径映射访问

#### 核心代码示例

```typescript
// src/security/sandbox.ts

import { execSync } from 'child_process';
import * as path from 'path';

export class Sandbox {
  /**
   * 在沙箱中执行命令
   */
  static execute(command: string, workDir: string, options?: {
    allowedDirs?: string[];
    network?: boolean;
    timeout?: number;
  }): { stdout: string; stderr: string; exitCode: number } {
    const allowedDirs = options?.allowedDirs || [workDir];
    const network = options?.network !== false ? '--share-net' : '--unshare-net';

    const bwrapArgs = [
      'bwrap',
      '--ro-bind', '/usr', '/usr',
      '--ro-bind', '/lib', '/lib',
      '--ro-bind', '/lib64', '/lib64',
      '--proc', '/proc',
      '--dev', '/dev',
      network,
    ];

    // 添加允许读写的目录
    for (const dir of allowedDirs) {
      const resolvedDir = path.resolve(dir);
      bwrapArgs.push('--bind', resolvedDir, resolvedDir);
    }

    bwrapArgs.push('--', '/bin/bash', '-c', command);

    try {
      const result = execSync(bwrapArgs.join(' '), {
        cwd: workDir,
        timeout: options?.timeout || 30000,
        encoding: 'utf-8',
      });
      return { stdout: result, stderr: '', exitCode: 0 };
    } catch (error: any) {
      return {
        stdout: error.stdout || '',
        stderr: error.stderr || error.message,
        exitCode: error.status || 1,
      };
    }
  }
}
```

### P3-4: 级联失败优化

**对应缺点**: #11 — failDependents 过于激进

#### 问题分析

当前 `TaskQueue.failDependents()` 将所有下游任务标记为 FATAL，即使某些下游任务与失败的上游任务无实际数据依赖。

#### 技术方案设计

引入"弱依赖"概念：
- **强依赖**（默认）：上游失败则下游必须失败
- **弱依赖**：上游失败不影响下游执行

在 Task 定义中添加 `weakDependencies` 字段。

#### 核心代码示例

```typescript
// src/types/index.ts — 修改 Task 接口

export interface Task {
  // ... 原有字段
  weakDependencies?: string[];  // 弱依赖列表
}

// src/task-queue/index.ts — 修改 failDependents

private failDependents(failedTaskId: string, error: TaskError): void {
  const failedTask = this.tasks.get(failedTaskId);
  if (!failedTask) return;

  for (const dependentId of failedTask.dependents || []) {
    const dependent = this.tasks.get(dependentId);
    if (!dependent || dependent.status !== 'pending') continue;

    // 检查是否为弱依赖
    const isWeakDep = dependent.weakDependencies?.includes(failedTaskId);
    if (isWeakDep) {
      console.log(`[TaskQueue] 任务 ${dependentId} 对 ${failedTaskId} 为弱依赖，跳过级联失败`);
      continue;
    }

    // 强依赖：级联失败
    this.failTask(dependentId, {
      type: 'FATAL',
      message: `Dependency "${failedTaskId}" failed: ${error.message}`,
      code: 'DEPENDENCY_FAILED',
    });
  }
}
```
